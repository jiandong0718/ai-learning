# 审查检查清单

按下面顺序执行 review：

1. 确认审查范围和变更来源
2. 先看 diff，再看相关实现、调用方和测试
3. 检查是否存在明确 bug 或行为回归
4. 检查边界条件、错误路径和状态时序
5. 检查测试是否覆盖关键路径和失败分支
6. 记录不合理假设、兼容性风险和未验证点
7. 按严重度输出 findings

## 优先看什么

- 会导致错误结果、崩溃、数据损坏或权限问题的改动
- 现有接口契约可能被打破的地方
- 新增条件分支、状态切换、并发处理、缓存和资源释放
- 测试没有覆盖但代码行为明显变复杂的部分

## 输出提醒

- 先写 findings，不要先写总结
- 能证明的问题直接下结论，不能证明的写清触发条件
- 没有明确问题时，也要说明剩余风险或测试缺口
