# 审查代码变更

## 概述

这个 skill 用于把“帮我 review 一下这次改动”转成稳定的审查流程。重点不是复述代码做了什么，而是优先找出会影响正确性、兼容性、可维护性和交付风险的问题，并按严重度先给 findings。

## 快速开始

1. 先确认 review 范围：diff、提交、分支或指定文件。
2. 基于变更定位相关代码、调用链、测试和配置。
3. 优先检查 bug、回归风险、边界条件和错误假设。
4. 再看遗漏测试、监控、兼容性和可维护性问题。
5. 输出时先列 findings，再补充开放问题、假设和简短总结。
6. 不替用户回退别人改动，也不扩散到审查范围外的文件。

## 工作流程

### 1. 明确审查边界

- 先确认用户要审的是哪些改动。
- 如果仓库里有其他人未提交或无关改动，只把它们当背景，不要回退。
- 如果缺少关键上下文会影响判断，先补看必要文件，而不是直接下结论。

### 2. 建立变更上下文

- 先看 diff，再看受影响实现、调用方、测试和配置。
- 关注改动是否打破现有契约，例如：
  - 输入输出格式
  - 错误处理路径
  - 状态更新时序
  - 并发、缓存或资源释放
- 结论要尽量基于代码和可见行为，不要凭印象推断。

### 3. 按优先级找问题

- 第一优先级：
  - 明确 bug
  - 行为回归
  - 崩溃、数据错误、状态错乱
  - 安全或权限风险
- 第二优先级：
  - 缺失关键测试
  - 边界条件未覆盖
  - 兼容性或升级风险
  - 不成立的隐含假设
- 第三优先级：
  - 可读性明显下降
  - 局部设计不一致
  - 会增加后续维护成本的实现细节

### 4. 验证审查判断

- 能从现有代码直接证明的问题，给出明确判断。
- 不能完全确认但风险较高的问题，写成风险并说明触发条件。
- 如果缺少测试或验证证据，不要把猜测写成确定事实。

### 5. 组织输出

- findings-first，按严重度排序。
- 每条 finding 尽量包含：
  - 位置
  - 问题是什么
  - 为什么构成风险
  - 在什么条件下会触发
- 如果没有发现明确问题，要直接说明“未发现明确 findings”，并补充剩余风险或测试缺口。

## 输出规则

- 先写 findings，再写开放问题或假设，最后才写简短总结。
- 不要先做大段改动概述，掩盖真正的问题。
- 不要把风格偏好包装成高优先级缺陷。
- 不要要求回退或覆盖不属于本次 review 范围的改动。
- 引用文件时保留关键路径和必要行号。
- 用户只说“review 一下”时，默认采用代码审查口径，而不是实现方案口径。

## 典型触发语句

- “帮我 review 这个分支”
- “看一下这次改动有没有 bug 或回归风险”
- “审一下这个 patch，重点看测试有没有漏”
- “从 code review 角度检查一下这些文件”
