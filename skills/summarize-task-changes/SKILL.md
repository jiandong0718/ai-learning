# 总结任务改动

## 概述

这个 skill 用于把“这次任务到底做了什么”整理成稳定、可复用的交付总结。重点不是泛泛复述，而是明确说明做了哪些操作、改了哪些文件、影响了哪些类或模块、改动了多少行代码，以及哪些结论已经验证、哪些只是当前判断。

## 快速开始

1. 先确认本次要总结的任务范围。
2. 先列出本次做过的关键操作。
3. 优先统计改动文件、增删行和受影响代码范围。
4. 能识别类或模块时，明确列出。
5. 区分已验证结果和未验证项。
6. 最后按固定结构输出总结。

## 工作流程

### 1. 明确总结范围

- 判断用户要总结的是：
  - 当前任务
  - 当前分支改动
  - 某次提交
  - 某组指定文件
- 如果范围不明确，优先按“当前任务实际改动”总结。
- 不要把历史无关改动混进本次结果。

### 2. 汇总操作过程

- 按真实执行顺序提炼关键操作，例如：
  - 读了哪些文件
  - 改了哪些实现
  - 补了哪些测试
  - 跑了哪些验证
  - 新增了哪些脚本或模板
- 操作摘要要偏动作，不要写空泛概述。

### 3. 统计改动范围

- 优先统计：
  - 改动文件数
  - 新增行数
  - 删除行数
  - 总改动行数
- 如果当前目录是 git 仓库，优先运行 `scripts/collect_change_stats.py` 获取统计。
- 如果不是 git 仓库，退回到手动基于本次已知改动文件做总结，并明确说明统计范围受限。

### 4. 识别类、模块和关键文件

- 优先列出实际改动涉及的类、模块、组件或主要文件。
- 如果代码语言本身不以类为主，要明确说明“本次改动以函数或模块为主”。
- 不要为了凑数，把文件里所有类都当作被改动。

### 5. 整理验证情况

- 明确写出：
  - 跑了哪些测试或命令
  - 验证结果是什么
  - 哪些内容没验证
- 不要把“理论上没问题”写成“已经验证通过”。

### 6. 组织交付结果

- 先给本次任务做了什么。
- 再给改动范围和统计数字。
- 再给受影响类、模块和关键文件。
- 最后给验证情况和剩余风险。

## 输出规则

- 不要只写“已完成某某任务”，要写清楚具体动作。
- 不要在没有统计依据时假装给出精确数字。
- 如果类名来自启发式识别，要明确说明可能不完整。
- 回答中保留关键文件路径、类名、命令和统计口径。
- 如果当前环境不是 git 仓库，要明确说明统计是手动汇总或受限统计。

## 典型触发语句

- “总结一下这次任务你都做了什么”
- “告诉我这次改了哪些类、多少个文件、多少行代码”
- “给我一份变更汇报，包含操作步骤和代码统计”
- “把这次开发任务的改动范围整理一下”
