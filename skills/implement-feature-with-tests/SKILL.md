# 实现功能并补测试

## 概述

这个 skill 用于把“帮我在这个项目里实现一个功能并保证质量”转成稳定的工程执行流程。重点不是只把代码写出来，而是先理解上下文，再做最小有效改动，并用测试或验证结果说明改动是成立的。

## 快速开始

1. 先明确需求边界和验收标准。
2. 先读相关代码，再决定改动点。
3. 优先做最小闭环实现，不要无关重构。
4. 同步补测试或更新已有测试。
5. 运行最小必要验证。
6. 输出改动结果、验证情况和剩余风险。

## 工作流程

### 1. 明确任务目标

- 判断用户要的是新增功能、修改行为、修复边界情况，还是补测试。
- 提炼可验证的验收标准。
- 如果信息不完整但仍可安全推进，先基于最合理假设实现，并在结果里写明假设。
- 如果关键缺失会直接影响实现方向，再请求补充，而不是带着高风险猜测编码。

### 2. 建立代码上下文

- 先定位入口文件、调用链、相关测试和配置。
- 优先理解现有模式，再决定是否复用。
- 记录会影响改动的约束，例如：
  - 类型系统
  - 错误处理约定
  - 测试框架
  - 命名和目录结构
- 不要一上来就改代码。

### 3. 设计最小改动方案

- 优先局部改动，避免顺手扩大范围。
- 只有在现有结构无法承载需求时，才扩展抽象。
- 如果要新增文件，说明它为什么必要。
- 如果存在多种实现路径，优先选：
  - 与现有代码风格一致
  - 依赖最少
  - 最容易验证

### 4. 实现功能

- 只修改与当前任务直接相关的代码。
- 保留现有接口契约，除非需求明确要求变更。
- 对不直观的逻辑可以加简短注释，但不要写低价值注释。
- 不要把功能实现和大规模重构混在一起。

### 5. 补测试和验证

- 优先补最接近改动行为的测试。
- 如果已有测试文件合适，优先在原位置扩展。
- 测试至少覆盖：
  - 主路径
  - 一个关键边界条件
  - 失败或异常分支（如果该行为重要）
- 能运行测试就运行；不能运行时明确说明原因和未验证范围。

### 6. 组织交付结果

- 先说明实现了什么。
- 再说明测试或验证结果。
- 最后说明仍然存在的风险、假设或未覆盖项。
- 用户如果只问“帮我改”，也不要省略验证结论。

## 输出规则

- 不要在没读上下文时直接生成大段代码。
- 不要为了显得完整而擅自重构无关模块。
- 不要漏掉测试影响分析。
- 不要把“理论上可行”当成“已经验证”。
- 如果没跑测试，要明确说没跑，并说明原因。
- 回答里保留关键文件路径、测试命令和必要的行为变化说明。

## 典型触发语句

- “帮我在这个仓库里实现这个功能，并补上测试”
- “基于现有代码改一下这个接口行为，顺便把测试补齐”
- “看一下这个需求应该改哪里，直接实现并验证”
- “在不大改结构的前提下，把这个功能接进去”
