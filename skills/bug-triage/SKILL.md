# Bug 排查与最小修复

## 概述

这个 skill 用于把“先帮我查清这个 bug 是怎么来的”转成稳定的工程排查流程。重点不是一上来就改代码，而是先复现或确认现象，缩小范围，识别最可能原因，再决定是只给结论，还是顺手实施一个边界清楚、容易验证的最小修复。

## 快速开始

1. 先确认 bug 现象、预期行为和影响范围。
2. 先找复现路径、日志、报错和最近相关代码。
3. 优先缩小范围，不要同时怀疑整个系统。
4. 区分已验证事实、推测原因和待确认项。
5. 能安全落地时实施最小修复，否则先交付定位结论。
6. 输出时写明验证方式、剩余风险和未确认部分。

## 工作流程

### 1. 明确问题定义

- 判断问题属于报错、功能异常、性能退化、状态错乱，还是回归问题。
- 提炼最小问题描述：什么场景下出现、实际结果是什么、预期是什么。
- 如果用户提供的信息不足以判断方向，优先补齐最关键的复现条件。
- 不要把“看起来像”直接当成结论。

### 2. 建立排查上下文

- 先定位相关入口、调用链、配置、测试和最近可能受影响的模块。
- 优先收集高信号证据，例如：
  - 报错栈
  - 日志
  - 复现步骤
  - 相关测试
  - 特征输入
- 如果问题疑似回归，优先看最近改动附近，而不是全局发散搜索。

### 3. 缩小范围并形成假设

- 先确认问题在哪一层出现：输入、状态流转、业务逻辑、边界条件、外部依赖或渲染表现。
- 每次只推进少量高价值假设，避免并行猜测太多分支。
- 对每个候选原因，尽量给出能证伪或证实的依据。
- 明确区分：
  - 已复现的问题
  - 已验证的原因
  - 仍在推测的原因

### 4. 决定是否实施最小修复

- 如果根因和改动边界已经足够清晰，优先做最小有效修复。
- 如果问题仍不稳定、信息不足或修复面过大，先交付排查结果和建议，不要冒进改动。
- 修复方案优先满足：
  - 改动范围小
  - 与现有模式一致
  - 易于验证
  - 容易回退

### 5. 验证结果

- 优先验证复现路径是否恢复正常。
- 能补测试就补最贴近问题的测试。
- 如果无法运行测试或无法完整复现，要明确说明卡点。
- 除了“问题消失”，还要检查是否引入明显副作用。

### 6. 组织交付

- 先给问题结论或当前最可能原因。
- 再写证据、改动或建议。
- 最后说明验证情况、剩余风险和下一步建议。
- 如果只是完成排查没有改代码，也要明确写出为什么暂不修改。

## 输出规则

- 不要在没确认问题范围前直接大改代码。
- 不要把猜测写成已定位结论。
- 不要为了排查方便顺手重构无关模块。
- 如果实施了修复，要保留关键文件路径、验证命令和行为变化说明。
- 如果没实施修复，要说明阻塞点、建议的下一步和风险等级。
- 用户问“哪里有问题”时，优先回答根因和证据，不要先堆方案。

## 典型触发语句

- “帮我排查这个 bug，看看最可能是哪一层出了问题”
- “这个报错是怎么来的，先定位原因再决定要不要改”
- “最近改完之后这里行为不对，帮我缩小范围”
- “看一下这个回归问题能不能直接做个最小修复”
- “分析一下异常现象，给出验证结论和风险”
