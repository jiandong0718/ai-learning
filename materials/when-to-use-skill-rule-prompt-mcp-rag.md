# 什么时候用 skill、rule、prompt、MCP、RAG

这份文档不是讲定义，而是帮你做判断。

## 一句话判断

- 想定义“这次要做什么”：用 `prompt`
- 想定义“哪些事绝对不能乱做”：用 `rule`
- 想定义“这一类任务一般怎么做”：用 `skill`
- 想让 agent 连上外部工具或系统：用 `MCP`
- 想让 agent 先取回相关知识再回答：用 `RAG`

## 常见判断题

### 1. “我每次都要重复解释一遍怎么查官方文档”

优先考虑：`skill`

原因：

- 这是重复任务
- 有稳定流程
- 有固定工具优先级
- 有固定输出要求

### 2. “我不希望 agent 编造来源”

优先考虑：`rule`

原因：

- 这是全局约束
- 不只在一个任务里成立
- 应该长期生效

### 3. “这次帮我看一下这个 bug”

优先考虑：`prompt`

原因：

- 这是当前一次任务
- 它决定本次目标
- 但如果 bug 排查反复出现，就可以再沉淀成 `skill`

### 4. “我需要 agent 去查公司的内部文档系统”

优先考虑：`MCP`

原因：

- 这里缺的是连接能力
- 没有接入，agent 根本拿不到数据

### 5. “我希望回答时参考知识库里最相关的内容”

优先考虑：`RAG`

原因：

- 这里缺的是检索增强
- 重点是取回相关知识，不是定义流程

## 组合方式

真实项目里，通常不是五选一，而是组合使用。

一个典型组合可能是：

- `rule`：不允许伪造结论，涉及最新信息必须先查
- `prompt`：帮我查某个接口现在怎么用
- `skill`：优先查官方文档，再提炼结论
- `MCP`：连接官方文档服务
- `RAG`：从团队私有知识库补充内部约定

## 什么时候应该把 prompt 升级成 skill

如果一个 prompt 同时满足下面 3 条，就很值得升级成 skill：

- 会重复出现
- 每次都需要差不多的步骤
- 输出质量依赖固定约束或固定资源

典型信号包括：

- 你总在复制同一段说明
- agent 每次处理方式都不一致
- 你总要反复提醒“先做 A，再做 B”
- 这类任务越来越依赖模板、脚本或参考资料

## 什么时候不该急着写 skill

下面这些情况，先别急着写：

- 任务只做一次
- 连边界都还没搞清楚
- 根本没有稳定流程
- 只是想保存一段 prompt
- 缺的其实不是流程，而是数据接入或知识来源

这时更适合先用：

- 临时 `prompt`
- 少量 `rules`
- `MCP` 接工具
- `RAG` 接知识库

## 最后一个判断标准

问自己一句话：

“我现在缺的，到底是目标、边界、流程、能力接入，还是知识补充？”

对应关系就是：

- 目标：`prompt`
- 边界：`rule`
- 流程：`skill`
- 能力接入：`MCP`
- 知识补充：`RAG`
