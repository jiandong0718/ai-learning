# Skill 质量检查表

这份清单用于判断一个 skill 是否“能用”，而不只是“看起来完整”。

## 一、触发是否清楚

- `name` 是否简短、具体、可辨认
- `description` 是否同时写清：
  - 做什么
  - 什么时候使用
- 看完 `description` 后，是否能想到 3 条真实触发语句
- 是否能和相邻 skill 区分开，不容易误触发

如果这部分做不好，skill 基本就不会被稳定用对。

## 二、流程是否可执行

- `快速开始` 是否给出了最短执行路径
- `工作流程` 是否按真实顺序写，而不是概念堆砌
- 每个步骤是否都带动作词，例如：
  - 先定位
  - 先验证
  - 再输出
- agent 进入 skill 后，是否能立刻知道第一步做什么

如果这部分做不好，skill 会变成“解释文档”，不是“执行方法”。

## 三、输出是否受约束

- 是否明确写了不要犯什么错
- 是否区分了事实、推断和未确认项
- 是否要求保留关键术语、文件路径、命令或来源
- 是否要求说明验证状态，而不是默认当作已验证

如果这部分做不好，输出会飘，质量也不稳定。

## 四、资源是否有用

- `references/` 是否真的在执行时能被读取和使用
- 模板文件是否能直接套用，而不是只有标题没有结构
- 检查清单是否短小、按顺序、可执行
- 是否把长资料放进 `references/`，而不是全部塞进 `SKILL.md`

如果资源只是“看起来有目录”，这个 skill 仍然是空壳。

## 五、边界是否明确

- 是否说明了适用场景
- 是否说明了不适用场景
- 是否避免和其他 skill 重叠过多
- 是否说明了哪些情况必须停下来说明，而不是继续脑补

边界不清，skill 就会和 prompt、rules 互相打架。

## 六、上下文是否节制

- `SKILL.md` 是否只保留核心流程
- 详细说明是否被拆到 `references/`
- 是否避免重复写同一件事
- 是否避免把通用常识写得太长

skill 不是越长越强，越短越准通常更好。

## 七、最终判断

一个合格 skill，至少应该满足下面 4 条中的 3 条：

- 能被正确触发
- 能指导实际执行
- 能稳定输出质量
- 能减少临时 prompt 和临场发挥

如果连这 4 条里的前 2 条都做不到，就应该重写，而不是继续补细节。
