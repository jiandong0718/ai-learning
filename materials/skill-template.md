# Skill 模板

这份模板不是教你“解释什么是 skill”，而是让你直接填空，尽快写出一个可用的 skill。

## 1. 先写一句定义

先用一句话回答：

- 这是什么类型的重复任务
- 什么时候应该触发它
- 结果要稳定成什么样

示例：

- 查询官方文档并给出来源依据
- 在现有代码库里实现功能并补测试
- 以 code review 视角审查变更并优先输出 findings

## 2. 先写 3 到 5 条真实触发语句

不要写抽象标签，要写用户真的会说的话。

示例：

- “帮我在这个仓库里实现这个功能，并补上测试”
- “审一下这个 patch，重点看回归风险”
- “先帮我定位这个 bug，再决定要不要改”

## 3. 写目录结构

最小结构：

```text
my-skill/
├── SKILL.md
```

常用结构：

```text
my-skill/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── references/
├── scripts/
└── assets/
```

## 4. 直接填 `SKILL.md`

```md
---
name: my-skill
description: [做什么]，[什么时候使用]，[结果应包含什么或优先满足什么]
---

# 标题

## 概述

这个 skill 用于把“[用户常见说法]”转成稳定的执行流程。重点不是 [常见误区]，而是 [真正目标]。

## 快速开始

1. 先 [第一步]。
2. 再 [第二步]。
3. 优先 [第三步]。
4. 然后 [第四步]。
5. 最后 [输出或验证]。

## 工作流程

### 1. 明确任务边界

- 判断这次任务属于 [任务类型 A / B / C]。
- 提炼最小目标、输入条件和验收标准。
- 如果信息不足但能安全推进，说明假设。
- 如果关键缺失会影响方向，再请求补充。

### 2. 建立上下文

- 先看 [代码 / 文档 / 资料 / 数据 / 配置]。
- 优先定位 [入口、调用链、相关测试、关键术语、来源]。
- 不要一开始就 [直接改代码 / 直接下结论 / 全局搜索]。

### 3. 执行核心动作

- 优先 [最小闭环动作]。
- 如果有多条路径，优先选：
  - 与现有模式一致
  - 最容易验证
  - 依赖最少
- 不要顺手扩大范围。

### 4. 验证或整理结果

- 验证 [主路径 / 边界条件 / 关键风险]。
- 如果无法完整验证，明确说明原因。
- 结果里区分：
  - 已确认事实
  - 推断
  - 未确认项

### 5. 组织交付

- 先给 [结论 / findings / 实现结果]。
- 再给 [依据 / 验证 / 关键改动]。
- 最后给 [风险 / 假设 / 下一步建议]。

## 输出规则

- 不要 [常见错误 1]。
- 不要 [常见错误 2]。
- 如果没做 [验证 / 查询 / 运行]，要明确说明。
- 回答中保留 [关键术语 / 文件路径 / 命令 / 来源]。

## 典型触发语句

- “[触发语句 1]”
- “[触发语句 2]”
- “[触发语句 3]”

## 资源

### references/

- 开始前先看 [checklist.md](references/checklist.md)。
- 输出时参考 [template.md](references/template.md)。

### scripts/

- 如果这类任务有重复执行逻辑，优先运行对应脚本。
```

## 5. 填 `agents/openai.yaml`

```yaml
interface:
  display_name: "面向人的短名称"
  short_description: "一句话描述这个 skill 的用途"
  default_prompt: "使用 $my-skill 帮我完成这类任务，并按既定结构输出。"

policy:
  allow_implicit_invocation: true
```

## 6. 什么时候需要 `references/`

适合放进 `references/` 的内容：

- 检查清单
- 输出模板
- 主题分类表
- 术语表
- 详细但会反复读的说明

不适合放进 `references/` 的内容：

- 冗长教程
- 和 skill 无关的背景材料
- 已经在 `SKILL.md` 里写过一遍的重复内容

## 7. 写完后立刻自查

至少检查这 5 件事：

1. `description` 是否同时写清“做什么”和“什么时候用”
2. `快速开始` 是否真能指导 agent 开始执行
3. `输出规则` 是否足够防止胡编、漏验证、漏边界
4. `references/` 是否真的能在执行时派上用场
5. 这个 skill 是否比“一段临时 prompt”更稳定
