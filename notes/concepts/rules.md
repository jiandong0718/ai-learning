# rules

## 是什么

在 AI agent 语境里，`rules` 是一组长期生效的行为约束，用来规定 agent 在执行任务时必须遵守哪些边界。

它通常不负责定义具体任务流程，而是负责限制：

- 什么不能做
- 什么必须优先做
- 哪些情况必须说明不确定性
- 哪些任务需要更高的准确性和验证标准

如果把 `prompt` 看成当前任务目标，把 `skill` 看成某类任务的做法，那么 `rules` 更像全局护栏。

## 为什么重要

如果没有 rules，agent 在很多任务里容易出现这些问题：

- 为了回答得完整而脑补事实
- 为了快而跳过验证
- 明明信息不足，却给出肯定结论
- 改动范围越界，顺手处理无关内容

rules 的价值在于：

- 给 agent 划清边界
- 统一不同任务下的行为底线
- 降低高风险任务中的不确定性
- 让输出更可控，而不是只靠临场发挥

## 在 AI 应用里怎么用

rules 适合放那些“跨任务都成立”的要求。

例如：

- 涉及最新信息必须先查证
- 不能伪造来源
- 修改代码前先读上下文
- 做 code review 时先报 findings
- 没跑测试时必须明确说明

在实际执行中，rules 通常和 prompt、skill 一起工作：

1. `prompt` 定义这次任务要做什么
2. `rules` 定义整个过程中哪些边界不能碰
3. `skill` 决定这类任务通常怎么做

所以 rules 更像“所有任务共享的约束层”。

## 常见误区

### 误区 1：rules 就是 prompt 的另一种写法

不是。

prompt 更偏当前任务的目标，rules 更偏长期约束。

### 误区 2：rules 越多越好

不是。

rules 太多会互相冲突，还会让 agent 变得僵硬。真正有价值的 rules 应该少而硬。

### 误区 3：有了 rules 就不需要 skill

不是。

rules 管边界，skill 管流程，两者不是同一种东西。

### 误区 4：rules 可以解决所有稳定性问题

不能。

如果流程本身不清楚、工具没接好、知识来源不可靠，单靠 rules 也无法保证结果稳定。

## 我的理解

我现在对 rules 的理解是：

- 它像 agent 的行为约束层
- 它更接近“底线”和“护栏”，不是“操作手册”
- 它特别适合放那些高风险、跨任务、必须长期生效的要求

之后继续学习时，我可以重点观察：

- 哪些要求应该写成 rules，而不是塞进 prompt
- 哪些 rules 真的是高价值约束
- 哪些规则其实应该下沉成 skill 流程，而不是做成全局限制
