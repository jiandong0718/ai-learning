# prompts

## 是什么

`prompt` 可以理解为模型在当前一次任务里接收到的指令和上下文，它定义的是“这次要做什么”。

它可以包含：

- 任务目标
- 背景信息
- 输出格式要求
- 输入材料
- 少量示例

在 AI agent 体系里，prompt 更偏一次性、面向当前请求，而不是长期复用的任务方法。

## 为什么重要

模型不是天然知道你这次到底要什么。

如果 prompt 不清楚，常见结果是：

- 任务目标模糊
- 输出格式不稳定
- 重点抓错
- 明明模型能力够，但结果还是不对路

prompt 的价值在于：

- 明确当前任务目标
- 提供本次任务所需上下文
- 约束本次输出结构
- 把抽象需求变成可执行请求

## 在 AI 应用里怎么用

prompt 最适合表达当前任务。

例如：

- “帮我查一下 OpenAI 官方文档里 Responses API 怎么做结构化输出”
- “帮我在这个仓库里实现一个功能并补测试”
- “把这段学习记录整理成每日复盘”

在真实应用里，prompt 常常会和其他层一起工作：

- 和 `rules` 配合：保证边界和行为底线
- 和 `skill` 配合：复用某类任务的稳定流程
- 和 `MCP` 配合：让模型能调用外部能力
- 和 `RAG` 配合：让模型先拿到相关知识再回答

所以 prompt 虽然重要，但它不是整个系统的全部。

## 常见误区

### 误区 1：prompt 写得越长越好

不是。

长 prompt 可能带来噪音、冲突和上下文浪费。好的 prompt 更强调清楚，而不是堆字数。

### 误区 2：prompt 可以替代 skill

不能完全替代。

如果一类任务会长期重复出现，而且有稳定流程，就更适合沉淀成 skill，而不是每次重新写 prompt。

### 误区 3：prompt 调好了，就不需要工程能力

不是。

很多稳定性问题来自工具接入、知识来源、验证机制和系统设计，不是只靠 prompt 就能解决。

### 误区 4：prompt 等于一句自然语言提问

不完全对。

在简单场景下可以是一句话，但在实际应用里，prompt 往往还包含格式约束、上下文、示例和状态信息。

## 我的理解

我现在对 prompt 的理解是：

- 它是当前任务的入口
- 它负责告诉模型“现在要干什么”
- 它适合表达一次性目标，不适合承载太多长期方法论

之后继续学习时，我可以重点观察：

- 哪些问题用一段清楚 prompt 就够了
- 哪些高频 prompt 应该升级成 skill
- prompt 中哪些信息是真必要的，哪些只是噪音
