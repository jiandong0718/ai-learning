# skills、rules、MCP、prompts、RAG 的关系

## 是什么

这几个概念都和 AI agent 怎么完成任务有关，但它们解决的不是同一个层面的问题。

- `prompt`：这次要做什么
- `rules`：做事时必须遵守什么边界
- `skill`：这一类事通常应该怎么做
- `MCP`：模型如何连接外部工具和外部数据
- `RAG`：模型如何在回答前先检索相关知识

可以把它们理解为一套分层协作结构，而不是互相替代的关系。

## 为什么重要

很多人一开始会把这些东西混在一起，结果常见问题是：

- 想用一段超长 prompt 同时解决目标、流程、边界、知识和工具接入
- 把 skill 写成知识文档，而不是任务方法
- 把 MCP 误解成某个搜索工具
- 把 RAG 误解成“只要接了知识库就会变聪明”

如果不把它们分层看，AI 应用会很容易出现：

- 指令重复
- 任务不稳定
- 工具调用混乱
- 回答看起来会说，但做事不可靠

## 在 AI 应用里怎么用

可以用一句话区分它们各自的职责：

- `prompt` 负责定义当前任务目标
- `rules` 负责定义行为约束
- `skill` 负责定义任务流程
- `MCP` 负责提供外部能力接口
- `RAG` 负责提供与当前问题相关的外部知识

把它们串起来看，常见执行链路像这样：

1. 用户发出一个请求，这形成当前 `prompt`
2. 系统或环境中的 `rules` 约束 agent 不能乱做什么
3. agent 判断是否应该触发某个 `skill`
4. 如果触发 skill，skill 会告诉 agent 优先读哪些资料、用哪些工具、按什么步骤执行
5. 需要外部工具时，通过 `MCP` 调用
6. 需要外部知识时，通过 `RAG` 或其他检索机制取回相关内容
7. 模型结合这些输入做推理、执行和输出

## 怎么区分

### prompt

`prompt` 是任务入口，偏一次性、面向当前请求。

例如：

- “帮我查一下 OpenAI 官方文档里 Responses API 怎么做结构化输出”
- “帮我在这个仓库里实现一个功能并补测试”

它定义的是本次任务，不一定可复用。

### rules

`rules` 是长期约束，偏全局性和强约束。

例如：

- 不能伪造来源
- 修改代码前先读上下文
- 回答 review 时优先报 findings
- 涉及最新信息必须先查证

它更像护栏，而不是操作手册。

### skill

`skill` 是可复用的方法包，偏某类任务的标准操作方式。

例如：

- 查询官方文档时先用官方文档 MCP
- 实现功能时先读代码、再做最小改动、补测试、给验证结果
- 做 code review 时先找风险和回归，不先写泛泛总结

它解决的是“这一类任务稳定怎么做”。

### MCP

`MCP` 是能力接入层，偏工具和数据源的标准化暴露。

例如：

- 文档查询服务
- 时间服务
- 记忆系统
- 数据库查询接口

它解决的是“agent 怎么连出去”。

### RAG

`RAG` 是知识增强方式，偏先检索再回答。

例如：

- 先从知识库取回和当前问题最相关的 5 段文档
- 再基于这些片段生成答案

它解决的是“模型怎么拿到和当前问题相关的外部知识”。

## 关系图

可以把它们想成这样：

- `rules` 决定边界
- `prompt` 决定当前目标
- `skill` 决定流程
- `MCP` 决定能调用哪些外部能力
- `RAG` 决定能补充哪些外部知识
- `model` 负责推理、判断和生成

如果要再压缩成一句话：

`prompt` 说现在要干什么，`skill` 说这类事一般怎么干，`rules` 说哪些事不能乱干，`MCP/RAG` 说你能拿到什么工具和知识。

## 常见误区

### 误区 1：skill 等于 prompt 模板

不是。

prompt 更像一次性指令，skill 是针对一类任务的长期方法包。

### 误区 2：MCP 等于 RAG

不是。

MCP 是标准化连接机制，RAG 是一种检索增强方法。RAG 可以通过 MCP 接入，但两者不是一回事。

### 误区 3：rules 越多越好

不是。

rules 太多会互相打架，还会让 agent 变得僵硬。rules 应该只写高价值、必须遵守的约束。

### 误区 4：有了 skill 就不需要思考

不是。

skill 只是降低不确定性，不会替代上下文理解、取舍判断和验证。

### 误区 5：RAG 解决一切知识问题

不是。

如果检索质量差、切片差、排序差，RAG 只会把噪音塞进上下文。

## 在 vibe coding 里的作用

在 vibe coding 里，这几个层次一起决定 agent 是“看起来会写”，还是“真的能稳定交付”。

`prompt` 让你快速表达需求，`skill` 把高频开发任务沉淀成固定套路，`rules` 防止 agent 越界乱改，`MCP` 让 agent 能查文档、读系统、接工具，`RAG` 让 agent 能拿到项目知识和团队知识。

如果只有 prompt，没有 skill 和 rules，结果通常是：

- 当次看起来能跑
- 下次做同类事风格就变了
- 容易漏测试、漏验证、漏边界

如果把 skill、rules、MCP、RAG 一起配好，vibe coding 才会从“临场发挥”变成“有工程手感的协作”。

## 我的理解

我现在更愿意把它们理解成一个 agent 操作系统里的不同层：

- `prompt` 是当前进程的输入
- `rules` 是系统级约束
- `skill` 是某类任务的执行模块
- `MCP` 是外设和服务总线
- `RAG` 是按需取回知识的检索层

之后继续学习时，我可以重点观察：

- 哪些问题应该靠 prompt 解决，哪些应该沉淀成 skill
- 哪些约束应该升级为 rules
- 哪些外部能力适合走 MCP
- 哪些知识适合做 RAG，而不是直接塞进 prompt
