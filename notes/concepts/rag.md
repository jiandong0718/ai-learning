# rag

## 是什么

`RAG` 是 Retrieval-Augmented Generation 的缩写，通常翻译为“检索增强生成”。

它的核心思路是：

1. 先根据当前问题去外部知识源检索相关内容
2. 再把检索结果提供给模型
3. 最后基于这些内容生成回答或执行结果

所以 RAG 解决的不是“模型怎么思考”，而是“模型在回答前怎么拿到相关外部知识”。

## 为什么重要

大模型本身存在几个天然限制：

- 知识可能过时
- 不掌握企业私有数据
- 容易在细节上幻觉
- 面对长文档时无法天然精确定位答案

RAG 的价值在于：

- 给模型补充最新或私有知识
- 降低纯靠记忆回答带来的幻觉风险
- 让回答更有依据
- 让知识库可以独立更新，不必每次都改模型

## 在 AI 应用里怎么用

一个典型 RAG 流程通常包含：

1. 准备知识源，例如文档、FAQ、内部资料、数据库记录
2. 做清洗、切分和索引
3. 根据用户问题做检索
4. 必要时重排结果
5. 把最相关内容连同用户问题一起交给模型
6. 输出回答，并在需要时给出引用

常见应用场景包括：

- 企业知识库问答
- 产品文档问答
- 客服辅助
- 代码库知识问答
- 内部流程和规范检索

在完整系统里，RAG 常和其他层配合：

- 和 `prompt` 配合：决定检索结果如何被使用
- 和 `MCP` 配合：通过标准化接口访问知识源
- 和 `skill` 配合：规定何时先检索、何时先用工具、何时给引用

## 常见误区

### 误区 1：接了知识库就自动变强

不是。

如果切分差、召回差、排序差，RAG 只会把不相关内容塞进上下文。

### 误区 2：RAG 可以完全消除幻觉

不能。

RAG 只能降低部分风险，不能保证模型一定正确理解和使用检索内容。

### 误区 3：RAG 就是向量数据库

不是。

向量数据库只是常见实现组件之一，RAG 是整套“检索 + 生成”的方法。

### 误区 4：所有 AI 应用都需要 RAG

不是。

如果问题本身不依赖外部知识，或者更适合直接调用工具、数据库或业务逻辑，RAG 可能不是最优解。

## 我的理解

我现在对 RAG 的理解是：

- 它像给模型加了一层按需取知识的机制
- 它特别适合解决“知识不在模型里”或“知识需要持续更新”的问题
- 它的关键不只是检索，而是检索结果是否真的能支撑后续生成

之后继续学习时，我可以重点观察：

- 文档应该怎么切分才更利于检索
- 检索结果怎样排序才更相关
- 什么时候该用 RAG，什么时候更适合 MCP 或直接工具调用
- 如何让回答带引用、可验证、可追溯
