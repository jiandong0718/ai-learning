# mcp

## 是什么

`MCP` 可以理解为模型上下文协议，核心作用是让模型以统一方式连接外部工具和外部数据源。

简单说：

- 模型本身负责理解任务和决定何时调用工具
- MCP 负责把外部能力标准化地暴露给模型

这些外部能力可以包括：

- 文档查询
- 网页抓取
- 时间查询
- 记忆系统
- 数据库或内部系统
- 本地或远程服务

## 为什么重要

大模型本身并不天然拥有最新数据、企业私有知识或任意系统操作能力。

如果没有 MCP，接入不同工具通常需要：

- 各写一套适配逻辑
- 各管一套输入输出格式
- 各自处理调用约定

MCP 的价值在于：

- 降低工具接入复杂度
- 统一模型与工具的协作方式
- 让工具更容易复用
- 让 agent 更容易扩展

## 在 AI 应用里怎么用

在 Codex 里，MCP 通常表现为配置好的 `mcp_servers`，然后暴露成可调用的工具。

常见形式有两种：

- `stdio`：本地通过命令启动服务
- `url`：远程通过网络连接 MCP 服务

例如一个配置可能长这样：

- `memory`：记忆能力
- `fetch`：抓取网页内容
- `time`：获取时间
- `openaiDeveloperDocs`：查询 OpenAI 官方开发文档

模型真正使用时，通常会以工具的形式出现，例如：

- `mcp__fetch__fetch`
- `mcp__context7__query_docs`

所以可以这样理解：

- `mcp server` 提供能力
- `tool` 是能力的调用入口
- `skill` 决定什么时候优先用这些能力

## 常见误区

### 误区 1：MCP 就是某一个工具

不是。

MCP 不是单个工具，而是一种统一接入工具和上下文资源的协议与机制。

### 误区 2：配置了 MCP 就一定能用

不是。

还可能受到这些因素影响：

- 当前会话是否已加载
- 本地依赖是否安装完成
- 网络是否可用
- 对应服务是否启动成功
- 当前代理是否暴露了对应工具

### 误区 3：MCP 只适合文档搜索

不是。

文档搜索只是常见场景之一。理论上任何能封装成标准接口的外部能力，都可以通过 MCP 接入。

### 误区 4：MCP 会替代应用工程

不会。

MCP 只是让接入更规范，不会自动帮你完成业务建模、错误处理、权限控制、缓存、监控和评测。

## 我的理解

我现在对 MCP 的理解是：

- 它像 AI 应用里的“工具总线”或“标准插座”
- 它解决的是模型如何稳定连接外部能力的问题
- 它让 agent 从“只会说”变成“能查、能取、能调用”

我之后继续学习 MCP 时，可以重点观察这几个问题：

- 一个 MCP server 暴露了哪些工具
- 输入输出结构是什么
- 哪些能力适合通过 MCP 接入
- 在真实项目里如何处理权限、失败重试和可观测性
