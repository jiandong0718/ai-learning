# skills

## 是什么

在 Codex 语境里，`skills` 是一类面向特定任务的能力说明包。它不是模型参数本身的一部分，而是一套围绕某种任务组织好的说明、流程和资源。

一个 skill 通常至少包含：

- `SKILL.md`
- 可选的 `references/`
- 可选的 `scripts/`
- 可选的 `assets/`

其中：

- `SKILL.md` 负责定义这个 skill 是什么、何时使用、如何执行
- `references/` 用于放详细参考资料
- `scripts/` 用于放可执行脚本
- `assets/` 用于放模板、图片、示例文件等资源

## 为什么重要

如果没有 skill，模型面对很多任务时只能临时组织思路，容易：

- 每次做法不一致
- 漏步骤
- 不知道优先用哪个工具
- 重复读取大量上下文

skill 的作用是把“这类任务怎么做”沉淀下来，让模型在遇到类似任务时更稳定、更高效。

## 在 AI 应用里怎么用

可以把 skill 理解为某一类工作的标准操作手册。

例如：

- 做 OpenAI 文档查询时，skill 可以规定先查官方 docs MCP，再决定是否回退到网页搜索
- 做某类数据处理时，skill 可以规定先运行哪个脚本，再读哪个参考文件
- 做公司内部任务时，skill 可以封装业务术语、数据库字段、流程要求

在实际执行中，skill 常见工作方式是：

1. 先通过 `name` 和 `description` 判断是否匹配当前任务
2. 如果匹配，再加载 `SKILL.md`
3. 根据 `SKILL.md` 决定读哪些 references、跑哪些 scripts、使用哪些工具

## 常见误区

### 误区 1：skill 就是插件

不完全对。

插件更偏“提供能力接口”，skill 更偏“提供任务方法和资源组织方式”。一个 skill 里可以包含插件或工具的使用说明，但 skill 本身不是单纯的工具接口。

### 误区 2：skill 越长越好

不是。

skill 应该尽量短，只保留核心流程。详细资料应该放到 `references/`，按需读取，否则会浪费上下文。

### 误区 3：skill 是给人看的文档

也不完全对。

skill 的第一读者是 AI agent，不是人类用户。所以重点不是写得像说明书，而是要让 agent 能稳定执行。

### 误区 4：有了 skill 就不需要 rules

不是。

skill 管“怎么做这类任务”，rules 管“必须遵守什么边界”。两者作用不同。

## 我的理解

我现在对 skill 的理解是：

- 它像“给 AI 的岗位 SOP”
- 它让通用模型在某个场景里表现得更像专业助手
- 它不是替代模型能力，而是降低任务执行的不确定性

我之后继续学习 skill 时，可以重点观察这几个问题：

- 这个 skill 解决的是哪类重复问题
- 它把哪些流程固化了
- 它依赖了哪些工具或资源
- 它有没有把上下文控制得足够精简
